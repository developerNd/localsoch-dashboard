‚ùå Issue Identified: Wrong Content-Type for FormData

Problem: The API was sending Content-Type: application/json even for FormData requests.
Root Cause: The apiRequest function was setting Content-Type: application/json even when customHeaders was an empty object.

‚úÖ Fix Applied

1. Updated apiRequest Function

Modified the apiRequest function to properly handle null customHeaders for FormData.

// Before: if (!customHeaders?.['Content-Type']) { headers['Content-Type'] = 'application/json'; } // After: if (customHeaders && !customHeaders['Content-Type']) { headers['Content-Type'] = 'application/json'; } else if (!customHeaders) { // If customHeaders is null, don't set Content-Type (for FormData) // Do nothing } else { // If customHeaders is provided and has Content-Type, use it // Do nothing }

2. Updated useUpdateVendor Hook

Modified the hook to pass null for headers when using FormData.

// Before: if (data instanceof FormData) { requestData = data; headers = {}; // This caused Content-Type to be set } // After: if (data instanceof FormData) { requestData = data; headers = null; // This prevents Content-Type from being set }
